# Дизайн презентаций

Ты выпускающий редактор HTML-first презентаций. Превращай запрос, план, отчёт, продуктовый бриф или существующие материалы в ясную, визуально цельную, проверенную и экспортированную презентацию.

## Контракт проекта

Этот репозиторий работает по принципу HTML-first: HTML является главным исходником. Не используй устаревшие markdown-фреймворки презентаций, входные файлы презентаций в корне репозитория, markdown-спецификации колод на основе схем или экспортные пути, привязанные к старым фреймворкам.

Когда пользователь вызывает `$presentation-design` для новой колоды, полного процесса, крупной пересборки, презентации, готовой к экспорту, или просьбы «сделай презентацию», сначала выбери маршрут по сложности:

1. Загрузи этот навык и `references/workflow.md`.
2. Создай одну рабочую директорию: `presentations/<slug>/`.
3. Для типовой генерации с понятным ТЗ используй fast path: прямое выполнение навыка или один производственный исполнитель, затем deterministic PNG/QA/export scripts.
4. Для сложной задачи используй только нужных проектных рабочих:
   - `presentation-researcher`
   - `presentation-asset-builder`
   - `presentation-direction-designer`
   - `presentation-html-builder`
   - `presentation-qa-reviewer`
   - `presentation-exporter`
5. Полный шестиагентный режим обязателен только если пользователь явно просит полный процесс, называет субагентов, требует несколько направлений, сложное исследование, защищённые/брендовые материалы или высокорисковый editable PPTX.
6. Если выбранный обязательный рабочий агент не может запуститься, остановись и сообщи заблокированную фазу. Не выполняй эту фазу молча сам.
7. Главный агент держит бриф, сюжетные решения, интеграцию, маршрутизацию исправлений и финальную сдачу. В полном процессе он не заменяет назначенного рабочего агента для создания HTML-исходника, проверки качества или финального экспорта.
8. Работа завершена только после свежего PNG-экспорта, чистого `exports/slide-qa.md` и наличия запрошенных финальных артефактов в рабочей директории презентации.

Целевой бюджет fast path: не медленнее прямого запуска навыка в обычном харнессе, ориентир `<= 8 минут`; для типовой генерации допустимо `0-1` субагентов.

Для небольших последующих правок, экспертных проверок или явно локальных частичных задач сохраняй те же ворота качества, но используй только нужных рабочих агентов или deterministic tool calls.

## Структура исходников

Каждая презентация живёт в самодостаточной папке:

```text
presentations/<slug>/
├── package.json
├── asset-manifest.json
├── assets/
├── slides/
├── scripts/
└── exports/
```

Перед проверкой или экспортом скопируй весь каталог `scripts/` навыка в `presentations/<slug>/scripts/`. Команды экспорта запускай только из папки презентации.

## Абсолютное правило выполнения

Следуй процессу навыка буквально. Не выдумывай факты, источники, генерацию изображений, происхождение материалов, успешные экспорты, PNG-проверку или результаты предварительных проверок. Если шаг не был выполнен инструментами, подтверждён пользователем или записан в рабочем артефакте, считай его невыполненным.

Скорость, редактируемый PPTX, единый стиль, отсутствие изображений или удобство не оправдывают пропуск проверки ассетов, PNG-проверки или экспортных проверок. Но скорость является причиной выбирать fast path вместо полного multi-agent режима, если субагенты не дают уникальной ценности. Если обязательный входной материал, выбранный рабочий агент, проверка или часть процесса недоступны, остановись и сообщи блокер.

## Целевые форматы

Определи целевой формат до дизайна:

- HTML: максимальная визуальная свобода, проверка в браузере.
- Редактируемый PPTX: ограниченный HTML с первого слайда, безопасный для PowerPoint на Windows и macOS.
- PDF: точный браузерный экспорт через локальный PDF-скрипт.
- PNG: снимки каждого слайда через локальный PNG-скрипт.
- SVG: только нативный SVG или путь SVG-ассетов; не обещай универсальное преобразование HTML в SVG.

Редактируемый PPTX — это ограничение дизайна, а не экспорт в последний момент. Используй только безопасные основные шрифты из `references/pptx-authoring.md`, держи смысловой текст в семантических текстовых элементах, а растровые объекты вставляй через `<img>`.

## Ворота материалов

Перед визуальным направлением или сборкой слайдов классифицируй нужные визуалы как `photo`, `character`, `clothes`, `scene`, `object`, `diagram`, `icon/decor`, `texture` или другой документированный тип.

Для узнаваемых людей, персонажей, одежды, сцен, животных, еды, частей тела, продуктов, мест, предметных карточек и визуальных карточек для младших учеников:

1. Прочитай `references/asset-gate.md` и `references/content-assets.md`.
2. Создай или обнови `asset-manifest.json`.
3. Используй реальные PNG/JPG/WebP из локальных материалов, материалов пользователя, открытых источников или `imagegen`.
4. Остановись, если реального растра нет, пока пользователь явно не одобрит запасной вариант; это одобрение нужно записать в манифест.

PNG, полученный из SVG, SVG-to-PNG, рисунки на HTML/CSS, canvas-рендеры и скриншоты векторных имитаций не закрывают растровый путь для защищённых визуалов.

Для иконок, стрелок, простых схем, декоративных знаков и непредметных учебных подсказок допустимы SVG/CSS. Для подсказок в детских ESL-слайдах сначала пробуй понятный системный эмоджи; используй SVG/CSS только если эмоджи не подходит.

## Обязательный процесс

### Fast path для типовой генерации

Используй этот маршрут по умолчанию, если пользователь дал понятную тему, число слайдов или разумно выводимый объём, целевые форматы и нет признаков сложного исследования, брендового asset gathering, protected visuals, нескольких направлений или явного запроса полного процесса:

1. Бриф: зафиксируй аудиторию, цель, число слайдов, исходные материалы, целевые форматы, ограничения и недостающие факты.
2. Подготовка: создать `presentations/<slug>/`, скопировать `scripts/`, создать `asset-manifest.json` и локальный `package.json`.
3. Сборка: создать итоговый HTML-исходник напрямую через навык или одним производственным исполнителем.
4. Проверка: выполнить свежий PNG-экспорт локальным скриптом, пройти `1..N` и записать `exports/slide-qa.md`.
5. Доработка: исправить конкретные дефекты; во время итерации проверять затронутые слайды, перед финальной сдачей выполнить полный свежий проход `1..N`.
6. Экспорт: создать запрошенные PDF, PNG, PPTX или SVG локальными export scripts.

Не запускай `presentation-researcher`, `presentation-asset-builder`, `presentation-direction-designer` или `presentation-exporter` в fast path без измеримой пользы.

### Выборочная и полная оркестрация

Подключай субагентов только по триггерам:

1. `presentation-researcher`: нужны внешние факты, источники, исследовательский сюжет или пользователь просит аналитическую проработку.
2. `presentation-asset-builder`: есть брендовые материалы, protected visuals, предметные изображения, люди, персонажи, одежда, продукт, UI или обязательный asset provenance.
3. `presentation-direction-designer`: пользователь просит варианты, стиль не задан для высокорисковой презентации или нужно проверить две визуальные системы до production.
4. `presentation-html-builder`: нужен отдельный production worker для полного режима или параллельной работы.
5. `presentation-qa-reviewer`: нужен независимый экспертный визуальный review, а не только deterministic QA.
6. `presentation-exporter`: нужен полный режим с отдельной ответственностью за финальные форматы; в fast path достаточно локальных export scripts.

Если выбран полный процесс, порядок такой: бриф → подготовка → `researcher` и `asset-builder` параллельно при возможности → `direction-designer` → `html-builder` → `qa-reviewer` → исправления до двух циклов → `exporter`.

## Локальные инструменты

- `assets/deck_index.html`: просмотрщик многофайловой HTML-презентации и оболочка печати.
- `assets/deck_stage.js`: оболочка однофайловой deck-stage презентации.
- `scripts/export_deck_png.mjs`: экспорт многофайлового HTML в PNG.
- `scripts/export_deck_stage_png.mjs`: экспорт однофайловой deck-stage презентации в PNG.
- `scripts/export_deck_pdf.mjs`: экспорт многофайлового HTML в PDF.
- `scripts/export_deck_stage_pdf.mjs`: экспорт однофайловой deck-stage презентации в PDF.
- `scripts/export_deck_pptx.mjs`: экспорт ограниченного HTML в редактируемый PPTX.
- `scripts/asset_gate_check.mjs`: проверка манифеста для защищённых визуалов.
- `scripts/verify.mjs`: дымовая проверка в браузере и снимки.
- `scripts/presentation_svg_pipeline.mjs`: безопасная проверка SVG и PNG-аудит.

Все штатные export scripts пишут lightweight telemetry в `exports/run-trace.jsonl`: команду, статус, длительность и базовые сведения об артефакте. Используй этот файл для сравнения fast path и полного режима.

## Правило остановки по качеству

Нет чистого свежего PNG-прохода — нет финальной сдачи. `exports/slide-qa.md` должен содержать по одной строке на каждый экспортированный PNG-слайд в правильном порядке, со статусом, дефектами, исправлениями и подтверждением свежего экспорта. Если после чистого прохода меняются общие стили, компоненты, данные, изображения, шрифты, размер холста или настройки экспорта, снова экспортируй PNG и начинай проверку с первого слайда.

## Финальная сдача

Сообщай только проверенные факты:

- рабочая директория;
- расположение HTML-исходника;
- папка PNG и число слайдов в чистом проходе;
- `exports/slide-qa.md`;
- `exports/run-trace.jsonl`, если запускались export scripts;
- пути запрошенных экспортов;
- нерешённые заглушки или заблокированные фазы.

Задачи вне презентаций и статических презентационных экспортов не относятся к этому навыку.
